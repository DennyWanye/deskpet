## 1. Executive Summary

P2-1 引入云端 LLM 作为本地 Ollama 的**能力补充**（非替代），同时上线第一版多 Persona 基线。这一 sprint 的关键架构收敛点是：**LLM Provider 统一为单一 `OpenAICompatibleProvider` 类**，本地 Ollama 通过其 OpenAI-compat 端点 (`/v1/chat/completions`) 接入，云端 DashScope 通过其兼容模式端点接入，两边对称。Persona 通过引用 "Provider Profile" 间接选择后端，而不是硬编码 provider 类型。默认策略为 `local_first`：任何请求优先尝试本地 profile，失败或 persona 显式指向云端 profile 时才走云。

本文档作为 8 个 slice (P2-1-S1 ~ P2-1-S8) 的总设计；每个 slice 实施前仍需通过 `sp-writing-plans` 产出独立 plan。

### 3.2 关键抽象（4 个）

| 抽象 | 文件 | 职责 |
|---|---|---|
| `OpenAICompatibleProvider` | `backend/providers/openai_compatible.py` | 用 OpenAI 协议格式（`POST /v1/chat/completions`，SSE 流）对任意兼容 endpoint 发请求。构造参数 `(base_url, api_key, model, ...)`；实现 `LLMProvider` Protocol 的 `chat_stream()`；在 init 时不持久化 api_key 明文。 |
| `ProviderProfile` | `backend/agent/provider_profiles.py` | 配置对象，3 字段 + 元信息：`{id, base_url, model, api_key_ref, kind: "local"\|"cloud", display_name}`。`kind` 显式标注，不靠 URL 启发式判断。 |
| `PersonaRegistry` | `backend/agent/persona_registry.py` | 管理 3+ persona 的注册表。每个 persona 拥有独立 `ServiceContext` 实例（复用 Phase 1 已有的 context）。 |
| `HybridRouter` | `backend/router/hybrid_router.py` | 根据策略（默认 `local_first`）+ persona 声明 + 预算状态，决定本次请求走哪个 profile；处理失败降级。 |

### 3.3 数据流：一次云端对话的路径

```
用户在 DialogBar 按下录音
  │
  ▼
Frontend → WebSocket /ws/audio → Backend
  │
  ▼
Backend pipeline: VAD → ASR → 得到文本 T
  │
  ▼
PersonaRegistry.current().service_context
  │ persona.provider_profile_id = "cloud_default"
  │ persona.provider_mode = "prefer_cloud"
  ▼
HybridRouter.route(
    persona=current_persona,
    text=T,
    budget=BillingLedger.month_remaining(),
)
  │
  ├─ policy == local_first
  ├─ persona.provider_mode == prefer_cloud  ← 显式走云
  ├─ budget > single_call_limit (¥1)
  ├─ budget > month_limit_remaining  → (80% 黄灯，100% 强降本地)
  ▼
Selected profile = "cloud_default"
  │
  ▼
secrets.get_api_key("cloud_default")  (Rust IPC)
  │ → 明文只在 process 内存中，不落盘
  ▼
OpenAICompatibleProvider(
    base_url="https://dashscope.../compatible-mode/v1",
    api_key="<from credential manager>",
    model="qwen3.6-plus"
).chat_stream(messages=[...])
  │
  ▼ (SSE tokens)
Pipeline → TTS → Frontend audio out

同时：BillingLedger.record(profile_id, usage.total_tokens, cost_rmb)
     ObservabilityStack.emit("llm_ttft_seconds", ...)
```

### 3.4 失败降级

| 失败情景 | 处理 |
|---|---|
| 本地 Ollama 进程未启动 / 11434 连接拒绝 | HybridRouter 在 ≤500ms 内 detected，若存在云端 profile 且预算够 → fallback 云端；否则 return 错误 toast `"本地 LLM 未启动，请检查 Ollama 服务"` |
| 云端 DashScope 5xx / 超时 (> 15s) | 若 policy == local_first 或 fallback 已启用 → 在 ≤30s 内切回本地继续响应（对齐 P2-G2 门） |
| 预算超限（单次或月度） | 单次超限：该请求直接走本地，用户感知 "这轮回答风格略不同"；月度超限：**所有**云端 profile 被标记 degraded，tray 图标加黄点 + SettingsPanel 显示红点 |
| apiKey 为空 / 401 | Provider 报错后 HybridRouter 标记 profile 为 `disabled_until_user_action`，fallback 本地；用户下次打开 SettingsPanel 时该 profile 旁显示 `⚠️ 认证失败` |
| Persona JSON schema 错误 | 启动时 PersonaRegistry 跳过坏文件并 log warning，不崩溃；UI 的 Persona 下拉仍可用（只少那一个） |

## 4. 配置数据模型

### 4.1 三层配置源（优先级从高到低）

1. **Windows Credential Manager** —— **仅存** `api_key`。Key 名规则：`deskpet:profile:<profile_id>`。
2. **`%APPDATA%/deskpet/personas/*.json`** —— 每个文件一个 persona。
3. **`%APPDATA%/deskpet/provider_profiles/*.json`** —— 每个文件一个 provider profile（baseUrl / model / 元信息，**不含 apiKey**）。
4. **硬编码默认**（Python backend 模块常量）—— first-run 时 materialize 到 2 + 3 的文件系统路径。

### 4.2 Provider Profile JSON 结构

文件命名：`<profile_id>.json`（例如 `local_ollama.json`、`cloud_default.json`）。

```json
{
  "id": "cloud_default",
  "display_name": "云端 DashScope",
  "kind": "cloud",
  "base_url": "https://dashscope.aliyuncs.com/compatible-mode/v1",
  "model": "qwen3.6-plus",
  "api_key_ref": "cloud_default",
  "request_options": {
    "temperature": 0.7,
    "max_tokens": 2048,
    "extra_body": {}
  },
  "created_at": "2026-04-15T12:00:00Z"
}
```

- `kind`: `"local" | "cloud"`。**显式标注**，不靠 URL 启发式判断；这影响 HybridRouter 路由 + 预算是否计入账单。
- `api_key_ref`: 指向 Credential Manager 的 profile id（通常 = `id`）。设 `null` 表示不需要 API key（本地 Ollama 即 null；Ollama 在 OpenAI-compat 模式下 api_key 字段实际会发送字符串 `"ollama"` 作占位，不验证）。
- `request_options.extra_body`: OpenAI SDK 的 escape hatch，用于透传 Ollama 特有参数（例如 `{"keep_alive": "10m"}`）。

**硬编码默认 2 个 profile**：

```python
DEFAULT_PROVIDER_PROFILES = [
    ProviderProfile(
        id="local_ollama",
        display_name="本地 Ollama",
        kind="local",
        base_url="http://localhost:11434/v1",
        model="gemma4:e4b",
        api_key_ref=None,
        request_options={"temperature": 0.7, "max_tokens": 2048, "extra_body": {}},
    ),
    ProviderProfile(
        id="cloud_default",
        display_name="云端 DashScope",
        kind="cloud",
        base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
        model="qwen3.6-plus",
        api_key_ref="cloud_default",
        request_options={"temperature": 0.7, "max_tokens": 2048, "extra_body": {}},
    ),
]
```

### 4.3 Persona JSON 结构

文件命名：`<persona_id>.json`（例如 `local_pet.json`、`online_assistant.json`、`code_helper.json`）。

```json
{
  "id": "online_assistant",
  "display_name": "联网助手",
  "system_prompt": "你是一个能联网查询、基于云端大模型的通用助手……",
  "provider_profile_id": "cloud_default",
  "provider_mode": "prefer_cloud",
  "model_override": null,
  "memory_scope": "per_persona",
  "voice_preset": "zh-CN-XiaoyiNeural",
  "created_at": "2026-04-15T12:00:00Z"
}
```

- `provider_profile_id`: 引用 4.2 中的 profile。
- `provider_mode`: `"prefer_local" | "prefer_cloud" | "strict_local" | "strict_cloud"`。影响 HybridRouter 是否允许降级。
- `model_override`: 非空则覆盖 profile 的 `model` 字段；允许某个 persona 临时用不同模型不修改 profile。
- `memory_scope`: `"per_persona" | "shared"`。决定 MemoryPanel 查询的 SQLite 过滤条件。
- `voice_preset`: TTS 声音；保留字段，P2-1 内 TTS 引擎不变，此字段为 P2-2 预留。

**硬编码默认 3 个 persona**：

| id | display_name | provider_profile_id | provider_mode | memory_scope |
|---|---|---|---|---|
| `local_pet` | 本地桌宠 | `local_ollama` | `strict_local` | `per_persona` |
| `online_assistant` | 联网助手 | `cloud_default` | `prefer_cloud` | `per_persona` |
| `code_helper` | 代码助手 | `cloud_default` | `prefer_cloud` | `per_persona` |

### 4.4 `billing_ledger` SQLite 表

```sql
CREATE TABLE billing_ledger (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  ts_utc       TEXT    NOT NULL,           -- ISO 8601
  profile_id   TEXT    NOT NULL,
  model        TEXT    NOT NULL,
  persona_id   TEXT    NOT NULL,
  tokens_in    INTEGER NOT NULL,
  tokens_out   INTEGER NOT NULL,
  cost_rmb     REAL    NOT NULL,           -- 预计算，基于 DashScope 公价表
  status       TEXT    NOT NULL            -- "success" | "failed_mid_stream" | "rejected_budget"
);
CREATE INDEX idx_billing_ts ON billing_ledger(ts_utc);
CREATE INDEX idx_billing_profile ON billing_ledger(profile_id, ts_utc);
```

- 成本基于 Provider Profile 中可配的 `pricing` 字段（默认 DashScope 公价表硬编码在 Python），允许用户按实际账单调整。
- 查询 API：`BillingLedger.month_to_date_rmb()` / `single_call_limit_ok(est_tokens) -> bool` / `month_limit_status() -> "ok" | "warning_80" | "capped"`。

## 7. 安全与隐私

- **apiKey 全链路约束**：
  - 永远**不**出现在 `%APPDATA%/deskpet/**/*.json` 任何字段
  - 永远**不**出现在 `billing_ledger` 表（该表只有 `profile_id`）
  - 永远**不**出现在 structlog 日志（structlog processor 加 `redact_api_key` 过滤器）
  - 永远**不**出现在 crash reports（`crash_reports/` 目录里的 dump 要 scrub）
  - 永远**不**明文通过 WebSocket 下发到前端（前端 SettingsPanel 永远拿到 `***` 占位，真实值只在提交保存时上行到 Rust 再写进 Credential Manager）
  - 明文只在两个地方短暂存在：(a) Python backend 进程内存，(b) 发往云端的 HTTPS request header
- **本地网络边界**：本 sprint 不改变 `DESKPET_DEV_MODE` / shared secret WebSocket 守卫。
- **审计**：所有云端调用都会进 `billing_ledger`，包含 `persona_id` 用于后续 "谁的会话调了多少云端" 自审。

## 9. 刻意不做的事（YAGNI）

- **多云厂 SDK**：仍通过通用 OpenAI-compat 协议接入；不为 DashScope / DeepSeek / OpenRouter 写 vendor-specific SDK 集成
- **Function / Tool Calling**：P2-1 仅 chat streaming；tool call 是 P2-3 的范围
- **JSON Mode / Structured Output**：同上，P2-1 不引入
- **Vision / Multimodal**：P2-3 范围
- **Persona 共享内存（`memory_scope == "shared"`）**：schema 支持，但 P2-1 的 PersonaRegistry 只实现 `per_persona`；`shared` 留作 P2-2+ 扩展
- **PromptTemplate 版本化 / A/B 测试**：P2-1 的 persona `system_prompt` 就是一个字符串字段，不搞模板引擎
- **API key 自动轮换**：用户手动 `[Rotate]` 按钮在 S3 内实现（走流程：清除旧 → 输入新 → 写 Credential Manager），**不**做定时轮换
- **云端调用缓存 / 去重**：即使同一句话连续问两次也走两次云；P2-1 不引入 response cache
- **多 apiKey / 多账号**：一个 profile 绑一个 apiKey；用户想切换账号 → 编辑 profile；不做账号池轮询
- **计费按真实账单对账**：`billing_ledger` 的成本基于本地公价表预估；与 DashScope 实际账单的对账留给用户手动核查（公价表会随 DashScope 官方调价更新）
