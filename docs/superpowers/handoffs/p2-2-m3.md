## What changed

### `backend/config.py`, `config.toml`
New `VoiceConfig` dataclass and `[voice]` TOML section:
```toml
[voice]
vad_threshold_during_tts = 0.65
min_speech_ms_during_tts = 400
tts_cooldown_ms = 300
```

Deliberately omitted three fields from the original plan (`always_on_mic`,
`vad_threshold_normal`, `min_speech_ms_normal`) — the first is frontend
state, the other two duplicate `[vad]`. Single source of truth for "normal"
values remains `[vad]`.

### `backend/pipeline/voice_pipeline.py`
- `__init__` accepts three new kwargs (all with sensible defaults for
  backwards-compat): `vad_threshold_during_tts`, `min_speech_ms_during_tts`,
  `tts_cooldown_ms`. Last two flow into `BargeInFilter(cooldown_ms=...,
  min_speech_during_tts_ms=...)`.
- TTS start → `self.vad.set_threshold(0.65)`. `finally` block restores
  `self._vad_threshold_normal` (captured from `vad.threshold` at init).
  Covered by tests for the interrupt path and the exception path.

### Discovered and fixed: M1 barge-in wiring gap

During Task 14 script design I noticed `process_audio_chunk` only evaluates
`BargeInFilter.should_allow()` on the `speech_start` event, passing
`current_speech_duration_ms()`. But silero-vad sets `_speech_start_ms =
_ms_counter` before any counter increment, so at that exact frame duration
is always 0. The TTS-phase gate `speech_ms >= 400` therefore never
opened — so the "fast" barge-in path was effectively dead during TTS.

User-visible consequence: interrupts only worked via the `speech_end`
path (user finishes speaking → cancel prior task), perceived delay
~750ms instead of the designed ~300ms post-VAD-confirm.

Fix (committed as part of Task 13, since they're one logical change
"make the M3 config values actually gate something"):

- Added `_barge_in_fired_for_current_speech` flag to VoicePipeline.
- `speech_start` event only sets this flag to `False` (rearm) and sends
  the `vad_event` payload; no barge-in attempt there.
- After the events loop, every frame re-evaluates: if TTS active, flag
  not set, and `current_speech_duration_ms() > 0`, call
  `BargeInFilter.should_allow()`. First frame that clears the 400ms
  threshold fires `tts_barge_in` exactly once, sets the flag, calls
  `on_interrupted()` + `interrupt()`.
- `speech_end` branch also rearms the flag (covers edge case).

### `scripts/perf/barge_in.py`
System-level smoke test beyond pytest. Requires a running backend with
`DESKPET_DEV_MODE=1`. Three sub-tests:

1. Latency — inject sustained speech mid-TTS, measure to `tts_barge_in`.
   Target p95 < 600ms raw wall-clock (includes 400ms `min_speech_during_tts`).
2. Short-burst rejection — 100ms noise must NOT barge.
3. Cooldown rejection — loud burst 200ms after `tts_end` must NOT barge.

Exit 0 = all pass / 1 = any fail. Suitable for CI.

## Decision records

### Why we fixed the M1 wiring gap inside Task 13 instead of spinning a Task 13.5

Without the per-frame re-evaluation, every `[voice]` config value we just
added is decorative — the gate path doesn't execute. Shipping M3 without
this would be false progress. 10-line fix, 3 new tests. Scope creep was
evaluated and deemed worth it; the alternative (`follow-up ticket`) would
mean users never notice the difference.

### Why perf script targets p95 < 600ms, not < 200ms

The architecture doc's G1 target (< 200ms) is "from VAD confirming
sustained speech to interrupt dispatched". End-to-end wall-clock
adds the `min_speech_ms_during_tts` window (400ms default). Perf script
measures raw wall-clock from speech injection, so 600ms is the honest
total including the required sustain window.

### Why `[voice]` and not extending `[vad]`

Two reasons:
1. `[vad]` holds provider-level tunables (`threshold`, `min_speech_ms`,
   `min_silence_ms`) that are raw VAD semantics. `[voice]` holds
   TTS-phase overrides, a higher-level concept. Mixing would confuse.
2. Future additions (`always_on_mic`, frontend tunables, etc.) land
   under `[voice]` cleanly without crowding `[vad]`.

### Why we didn't add per-event barge-in rate limiting beyond "once per segment"

The `_barge_in_fired_for_current_speech` flag (reset on both speech_start
AND speech_end) already guarantees at most one barge-in per continuous
speech segment. Any additional limiter would just add state without
addressing a real behavior.

## Known limitations / follow-ups

1. **No frontend-side threshold feedback** — if user edits
   `[voice].vad_threshold_during_tts` in `config.toml`, they must restart
   the backend. A `POST /config/voice` hot-swap endpoint (mirroring
   `POST /config/cloud`) would be natural but is out of M3 scope.
2. **Speaker-echo failure mode** depends heavily on speaker + room
   acoustics. The 0.65 default was sized for a standard laptop setup;
   users with louder speakers or quiet mics may need the 0.75 fallback.
3. **Task 14 perf script requires live backend + DESKPET_DEV_MODE=1**.
   Running it from CI would need a backend harness. Not blocking.
4. **tag `p2-2-verified` is deferred** until Task 15 manual testing passes.
